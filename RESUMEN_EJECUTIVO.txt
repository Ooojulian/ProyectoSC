═══════════════════════════════════════════════════════════════════════════════
                              CARACTERÍSTICAS
═══════════════════════════════════════════════════════════════════════════════

✨ GENERADOR DE EVENTOS
  ├─ Crea eventos aleatorios (sensor simulado)
  ├─ Empaqueta en JSON con timestamp y origen
  ├─ Envía por UDP broadcast a toda la red
  └─ Configurable: GEN_RATE de 0.1 a 100 eventos/segundo

✨ RECEPTOR CONCURRENTE
  ├─ Escucha en thread separado sin bloqueos
  ├─ Filtra duplicados por ID único
  ├─ Simula interferencia: DROP_PROB (0-100%)
  └─ Procesa simultáneamente emisión y recepción

✨ PERSISTENCIA
  ├─ Guarda automáticamente en CSV local
  ├─ Sincronización distribuida (cada nodo = copia)
  ├─ Thread-safe, sin corrupción de datos
  └─ Recuperable después de restart

✨ MÉTRICAS (MODELO M/M/1)
  ├─ τ (Latencia): ms desde creación a almacenamiento
  ├─ ρ (Utilización): carga del sistema (0-1)
  ├─ W(q): tiempo promedio en cola
  ├─ L(q): paquetes en cola
  ├─ PLR: tasa de pérdida
  └─ Throughput: bytes/segundo

✨ INTERFAZ WEB
  ├─ Dashboard en cada nodo (independiente)
  ├─ Actualización automática cada 2 segundos
  ├─ Tabla de inventario sincronizado
  ├─ Topología de red visible
  ├─ Logs en vivo
  └─ Responsive (funciona en móvil)

═══════════════════════════════════════════════════════════════════════════════
                              PARÁMETROS AJUSTABLES
═══════════════════════════════════════════════════════════════════════════════

GEN_RATE=0.5    → Muy lento (1 evento cada 2 segundos)
GEN_RATE=1.0    → Normal (1 evento/segundo) ← DEFAULT
GEN_RATE=10.0   → Moderado (10 eventos/segundo)
GEN_RATE=50.0   → Estrés (50 eventos/segundo)
GEN_RATE=100.0  → Saturación (100 eventos/segundo)

DROP_PROB=0.0   → Sin pérdidas ← DEFAULT
DROP_PROB=0.1   → 10% de paquetes se pierden
DROP_PROB=0.3   → 30% de paquetes se pierden
DROP_PROB=0.5   → 50% de paquetes se pierden

PAYLOAD_SIZE=0       → Datos mínimos ← DEFAULT
PAYLOAD_SIZE=1000    → 1KB extra por paquete
PAYLOAD_SIZE=10000   → 10KB extra por paquete

CATALOG_SIZE=10      → 10 items únicos posibles ← DEFAULT
CATALOG_SIZE=100     → 100 items únicos posibles

                            MÉTRICAS DE RENDIMIENTO
═══════════════════════════════════════════════════════════════════════════════

ESPERADO CON CONFIGURACIÓN NORMAL (GEN_RATE=1.0, DROP_PROB=0.0):

  Latencia promedio    : 1-5 ms
  Throughput           : 1000-5000 paquetes/minuto
  CPU usage            : < 20%
  Memoria por nodo     : 50-100 MB
  Sincronización CSV   : 100% (todos iguales)

CON CARGA ALTA (GEN_RATE=50):

  Latencia promedio    : 10-50 ms
  Throughput           : 50000-100000 paquetes/minuto
  CPU usage            : 40-60%
  ρ (utilización)      : 0.8-0.95 (antes de saturación)
  Si ρ > 0.99          : Sistema saturado, algunas pérdidas

CON INTERFERENCIA (DROP_PROB=0.2):

  Paquetes perdidos    : ~20% de total enviados
  PLR en CSV           : ~0.2 (20%)
  Sincronización       : Aún 100% (deduplicación automática)
  Recuperación         : Automática, sin intervención

═══════════════════════════════════════════════════════════════════════════════
                        CONCEPTOS DE SISTEMAS COMPLEJOS
═══════════════════════════════════════════════════════════════════════════════

El proyecto demuestra estas propiedades de sistemas complejos:

1. EMERGENCIA
   → Comportamiento global emerge de interacciones locales
   → Sin coordinador central, el sistema se auto-sincroniza
   → La consistencia de datos emerge de broadcasts descentralizados

2. AUTO-ORGANIZACIÓN
   → BATMAN recalcula rutas automáticamente
   → Topología se adapta a cambios del ambiente
   → No hay nodo "central" que controle todo

3. ADAPTACIÓN
   → Sistema se recupera de nodos caídos
   → Redirige tráfico automáticamente
   → Continúa funcionando mientras haya mínimo 1 nodo

4. NO LINEALIDAD
   → Performance NO escala linealmente con nodos
   → Latencia aumenta exponencialmente con carga
   → Punto de saturación es predecible (ρ ≈ 0.9)

5. RESILIENCIA
   → Tolerancia a fallos simultáneos
   → Tolerancia a interferencia (DROP_PROB)
   → CSVs se sincronizan incluso con pérdidas
